# День 2: Архітектура та вибір технологічного стеку

**Дата:** [[2026-01-23]]
**Мета:** Розробити високорівневу архітектуру системи та обрати основні технології для розробки MVP.

---

## 1. Завдання на день

- [ ] **Визначити архітектурний підхід:**
  - [ ] Моноліт чи мікросервіси? (Для MVP краще почати з моноліту, щоб прискорити розробку).
  - [ ] Намалювати базову блок-схему системи: клієнт (веб-додаток), сервер (API), база даних, зовнішні сервіси (YouTube API, платіжна система).
- [ ] **Вибір Open Source проекту для основи:**
  - [ ] Пошук існуючих LMS (Learning Management System) з відкритим кодом.
  - [ ] Критерії пошуку: активна спільнота, хороша документація, сучасний технологічний стек, ліцензія (MIT, Apache 2.0).
  - [ ] Кандидати: Moodle, Open edX, Canvas LMS, або менші проекти на GitHub.
  - [ ] **Рішення:** Обрати 1-2 фінальних кандидати для більш глибокого аналізу.
- [ ] **Вибір технологічного стеку (якщо робимо з нуля):**
  - [ ] **Frontend:**
    - [ ] Фреймворк: React (Next.js), Vue (Nuxt.js), чи Angular? (Next.js - популярний вибір для SEO та швидкого старту).
    - [ ] UI-бібліотека: Material UI, Tailwind CSS, Bootstrap?
  - [ ] **Backend:**
    - [ ] Мова/платформа: Node.js (Express/NestJS), Python (Django/FastAPI), PHP (Laravel).
    - [ ] База даних: PostgreSQL (добре підходить для реляційних даних) чи MongoDB (якщо структура даних буде дуже гнучкою).
  - [ ] **Інше:**
    - [ ] Деплоймент: Docker, Vercel/Netlify (для фронтенду), DigitalOcean/AWS.

## 2. Результат дня

- Створено блок-схему архітектури.
- Складено короткий список (1-2) потенційних open-source проектів для використання.
- Сформовано пропозицію по основному технологічному стеку (напр., Next.js + NestJS + PostgreSQL).

## 3. Нотатки

- Важливо враховувати досвід команди (Андрія та мій) при виборі технологій.
- Якщо обираємо open-source проект, потрібно оцінити, наскільки легко буде інтегрувати наші унікальні фічі (форум, опитування, аналітика) та кастомізувати його.
- Потрібно одразу думати про те, як буде відбуватися деплоймент на сервери замовника. Використання Docker може значно спростити цей процес.
