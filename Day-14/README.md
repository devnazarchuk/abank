# День 14: Підготовка до розгортання та фінальний огляд

**Дата:** [[2026-02-04]]
**Мета:** Підготувати проект до розгортання на "бойовому" сервері, провести фінальний огляд та передати артефакти.

---

## 1. Завдання на день

- [ ] **Підготовка до деплойменту:**
  - [ ] **Конфігурація середовища:**
    - Створити `.env.production` файл з налаштуваннями для бойового середовища (URL бази даних, ключі API для Stripe, секретний ключ для JWT).
    - **УВАГА:** Переконатись, що `.env` файли не потрапляють в Git. Створити `.env.example` з переліком необхідних змінних.
  - [ ] **Docker-ізація (якщо обрано цей підхід):**
    - Написати `Dockerfile` для фронтенду та бекенду.
    - Написати `docker-compose.yml` для запуску всього стеку (додаток, база даних, nginx) однією командою.
  - [ ] **Скрипти для білду та запуску:**
    - В `package.json` додати/перевірити скрипти: `build` (для створення продакшн-білду) та `start` (для запуску зібраного додатку).
- [ ] **Документація:**
  - [ ] Оновити `README.md` в корені проекту (`Project-Code`).
  - [ ] Додати чіткі інструкції по розгортанню проекту (кроки для встановлення залежностей, збірки та запуску).
  - [ ] Описати основні змінні середовища, які потрібно налаштувати.
- [ ] **Фінальний огляд та передача:**
  - [ ] Провести останній огляд коду, переконатись, що в `develop` гілці знаходиться стабільна версія.
  - [ ] Створити `release` тег у Git (напр., `v0.1.0-mvp`).
  - [ ] Зробити злиття `develop` в `main` гілку.
  - [ ] Підготувати архів з кодом або надати доступ до Git-репозиторію.
- [ ] **Планування наступних кроків:**
  - [ ] Скласти список ідей та завдань для наступної ітерації розробки (напр., чат-бот, гейміфікація, розширені налаштування профілю).

## 2. Результат дня

- Проект повністю готовий до розгортання на сервері замовника.
- Створена детальна документація по запуску та налаштуванню.
- Git-репозиторій впорядкований, основні гілки оновлені.
- Сформовано бачення подальшого розвитку продукту.

## 3. Нотатки

- Розгортання — це окремий процес, який може зайняти час. Мета цього дня — максимально спростити цей процес для людини, яка буде цим займатися (навіть якщо це будете ви).
- Чітка документація є запорукою того, що проект можна буде легко підтримувати та розвивати в майбутньому.
- Цей день завершує двотижневий спринт по розробці MVP. Це чудова нагода, щоб ще раз обговорити результати з командою та зафіксувати досягнення.
